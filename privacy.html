Privacy Policy

App: Phone Monitor  ·  Last updated: 22 March 2026

Important: Phone Monitor is a device activity logging tool. It is designed to collect detailed information about how a device is used. Before installing this app on any device, ensure that the person using that device is aware it is installed and understands what is being recorded. Installing monitoring software on a device without the knowledge of the person using it may be illegal in your jurisdiction.
Short version: Phone Monitor records device activity events and stores them locally. No data is personally identifiable on its own — every event is associated only with a random cryptographic identifier, not with any name, account, or real-world identity. If you configure a server URL, data is uploaded there. The app developer has no access to that server and never receives your data. For complete control over your data, you can self-host the server.

1. What Phone Monitor does

Phone Monitor is a device activity logger intended for parental monitoring or personal activity tracking. It runs as a persistent background service and records events about how the device is used. All recorded data is stored in a local database on the device. Optionally, data can be uploaded to a server of your choosing for remote review.

The developer of this app does not operate any server that receives your data. The server is configured entirely by you. It may be a server you host and control yourself, or one operated by a third party you trust.

2. A note on identifiability

Every event recorded by Phone Monitor is associated with a device identifier — a random cryptographic value derived from a key generated in the device's secure hardware (Android Keystore). This identifier:

  • Is not your name, email address, phone number, Google account, or Android advertising ID.
  • Cannot be reverse-engineered to reveal who you are or which device you own.
  • Is not shared with any third-party service.
  • Is unique to the app installation and changes if the app is uninstalled and reinstalled.

This means that all data stored or uploaded by Phone Monitor is a series of timestamped activity events linked to an opaque random identifier — not to any personally identifiable information. The operator of the server you configure may be able to link events to a specific device, but only if they also have independent knowledge of which device that identifier belongs to.

3. Data collected — in detail

The following sections describe each category of data the app records. For each, we explain exactly what is and is not collected, and why the data is needed.

App interactions (app usage)

Collected:

  • The package name (e.g. com.example.app) and display name of every app that enters or leaves the foreground, with the exact timestamp and session duration.
  • A periodic summary (approximately every 5 minutes) of total foreground time per app.
  • Timestamps of screen-on, screen-off, and screen-unlock events.

Not collected: Any content displayed within those apps, keystrokes, search terms, or any interaction within an app beyond the fact that it was open.

Why this is required: This is the core purpose of the app — to provide a record of which apps are used, and for how long, to support parental oversight or personal activity review.

Installed apps

Collected:

  • The package name and display name of any app that is installed, updated, or uninstalled, along with a timestamp.

Not collected: A full list of all apps on the device is not periodically scanned or transmitted. Only individual install, update, and uninstall events are recorded as they happen.

Why this is required: Knowing which apps are added or removed from a device is an important part of device oversight — for example, to detect when new apps are installed by the device user.

Device diagnostics (battery)

Collected:

  • Battery charge level (percentage), approximately every 5 minutes.
  • Events when charging starts, charging stops, or the battery reaches a low threshold.
  • Where supported by the device hardware: instantaneous current draw (in microamperes) and remaining charge capacity (in microampere-hours).

Not collected: No other device hardware data, network diagnostics, or system performance metrics.

Why this is required: Battery data provides useful context for activity logs — for example, distinguishing periods of device inactivity from periods where the device was simply off or uncharged.

Device identifier

Collected:

  • A stable identifier derived from a cryptographic key pair generated inside the device's Android Keystore secure hardware. This value is a SHA-256 hash of the public key and is used to authenticate uploads to the configured server.

Not collected: This identifier is not the Android advertising ID (AAID), IMEI, IMSI, MAC address, SIM serial number, phone number, Google account ID, or any other system or hardware identifier. It is a randomly generated value that exists solely within this app.

Why this is required: The server needs a way to verify that uploaded data comes from a known, authorised device and has not been tampered with in transit. The cryptographic identifier serves this purpose without revealing anything about the device owner's real-world identity.

SMS receipt events

Collected: A timestamp recording that an SMS message was received. This requires the READ_SMS permission.

Not collected: The message content, the sender's phone number or name, the recipient, or any other information about the message. The app detects only that the inbox changed — it reads nothing from the message itself.

Why this is required: SMS receipt events are recorded so that incoming communication can be noted in the activity log. Only the timing of the event is retained, not its substance.

If the READ_SMS permission is not granted, no SMS events are recorded.

Notification events

Collected:

  • The package name of the app that posted or dismissed a notification, and a timestamp.
  • Whether the notification was "ongoing" (persistent) or dismissable.
  • A numeric hash of the internal notification key (a non-reversible integer, not the notification content).

Not collected: The notification title, message text, or any other content. These fields are present in the Android notification but are explicitly ignored by the app.

Why this is required: Notification events indicate when apps are actively communicating with the user (e.g. a messaging app receiving a message, or an alarm firing), providing useful context in the activity log without capturing the content of those communications.

This feature requires the notification listener permission, which must be granted manually in Android Settings.

4. Data explicitly not collected

The following is a non-exhaustive list of data the app does not record under any circumstances:

  • Your name, email address, or any account credentials
  • Your location or GPS coordinates
  • Camera or microphone input
  • Phone numbers (caller ID, dialled numbers, SMS sender)
  • Message content (SMS, notifications, or any in-app communication)
  • Contacts, photos, videos, or any files on the device
  • Web browsing history or clipboard content
  • Passwords or any on-screen content
  • The Android advertising ID (AAID) or any hardware identifier (IMEI, MAC, IMSI, etc.)

The developer does not operate any analytics service, crash reporter, or advertising network within this app. No data of any kind is sent to the app developer.

5. Where your data goes — and how to keep full control

All recorded events are stored in a database on the device. If no server URL is configured, no data ever leaves the device.

If you configure a server URL, events are uploaded in encrypted batches to that server. The upload is authenticated using the device's cryptographic key, so only events from registered devices are accepted. Data is transmitted over HTTPS and signed to protect integrity in transit.

For complete control over your data, self-host the server. Phone Monitor's server software is open source and designed to be run on your own hardware or cloud instance. If you self-host, your data never passes through any infrastructure you do not own or control. The app developer has no access to any self-hosted server and cannot view, retrieve, or delete your data.

If you choose to use a server operated by a third party, that operator's own privacy policy governs how they store and use your data once it is received. You are responsible for choosing a server operator you trust.

Once data is uploaded to any server, the app developer has no access to it, no visibility into it, and no ability to delete it. You should treat any data that leaves the device as permanently accessible to whoever operates that server.

6. Android permissions explained

  • READ_CALL_LOG / READ_PHONE_STATE — used to detect call state changes (ringing, in-call, idle). No call log entries or phone numbers are read.
  • READ_SMS — used to detect when an SMS arrives via a content observer on the inbox. Message content is not read.
  • POST_NOTIFICATIONS — required on Android 13+ to display the persistent status notification while the service is running.
  • PACKAGE_USAGE_STATS — required to identify which app is in the foreground. Must be granted manually in Android Settings.
  • BIND_NOTIFICATION_LISTENER_SERVICE — required to observe notification arrivals and dismissals. Must be granted manually in Android Settings. Title and content are not read.
  • FOREGROUND_SERVICE — required to keep the monitoring service running reliably in the background.
  • RECEIVE_BOOT_COMPLETED — allows monitoring to resume automatically after the device restarts, if it was active before.
  • INTERNET / ACCESS_NETWORK_STATE — required to upload data to a configured server.
  • REQUEST_IGNORE_BATTERY_OPTIMIZATIONS — allows the app to request exemption from battery optimisation, improving reliability of the background service.

7. Data retention and deletion

On the device, events are deleted from local storage immediately after they are successfully uploaded to the configured server. If no server is configured, events accumulate in the local database until the app is uninstalled.

Uninstalling the app removes the local database entirely, including all events that have not yet been uploaded.

Data that has already been uploaded to a server cannot be deleted by this app. You must contact the operator of that server directly to request deletion. If you self-host the server, you have full control to delete any data at any time.

You are solely responsible for ensuring your use of this app complies with applicable laws in your jurisdiction, including any laws relating to consent, workplace monitoring, and electronic surveillance. Do not install this app on a device belonging to another person without their knowledge.

9. Children's privacy

This app may be used by parents or guardians to monitor a minor child's device. In such cases, the parent or guardian is the party responsible for the data collected and for any server it is sent to. The app developer does not receive, store, or process data about anyone — child or adult.

The data collected by this app (described in Section 3) is the same regardless of the age of the device user. No additional categories of data are collected when the device is used by a child, and no data is shared with any advertising or analytics provider.
